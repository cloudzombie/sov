Genesis Ceremony — Tax Account Binding

Generate the founder (5%) & dev (2%) tax accounts, hand over only the public keys, and bind them into block 0. Confirm each step as you complete it — progress is saved on this device.

Non-negotiable principle. The seeds are generated by you and never touch the assistant, the repo, or the network. You share only each account's public key (safe to publish on-chain). Whoever holds a mnemonic owns that tax stream forever — treat the 24 words like the crown jewels. The tax accounts must be key-bound in genesis: a named account left keyless is claimable by the first RotateKey to reach it, so binding your keys at block 0 closes that window before a single coin is minted.

Generate the two accounts (offline)

you only▶

Build the wallet once, then generate two separate accounts with two independent seeds. This is fully offline — nothing is sent anywhere.

cd chain && cargo build --release
./target/release/sov-wallet new founder.tax.sov # the 5% account ./target/release/sov-wallet new dev.tax.sov # the 2% account

Each run prints — for your eyes only — a fresh 24-word mnemonic and the keys it derives (hybrid Ed25519 + ML-DSA-65, post-quantum by default):

Write each mnemonic down on paper, store the two separately. For a forever (mainnet) account, run this on an air-gapped / freshly-booted machine and never type either mnemonic into an internet-connected host except to sign. You can confirm recovery any time with sov-wallet import "<mnemonic>" — it must reprint the same public_key.

3

Assistant binds them into genesis

assistant▶

With your two public keys (no seeds involved) the assistant adds two key-bound, zero-balance accounts to chain/specs/testnet-1.json. Zero balance keeps the no-pre-mine guarantee — the keys only establish control; the balance accrues purely from the tax.

{ "account": "founder.tax.sov", "public_key": "hybrid65:0x…", "balance": 0 }, { "account": "dev.tax.sov", "public_key": "hybrid65:0x…", "balance": 0 }
  • Adding genesis accounts changes the genesis hash → the assistant re-pins the determinism KAT and re-runs it.
  • MiningPolicy already names founder.tax.sov (500 bps) / dev.tax.sov (200 bps), so no Rust change is needed unless you want different names.

Relaunch & watch the tax arrive

5

Prove you control them (spend test)

you only▶

Seeing a balance is not control. Confirm you can spend the tax with your mnemonic — the node accepts the transfer only if the signature matches the key bound in genesis. (Run this where it's safe to type the mnemonic.)

./target/release/sov-wallet 127.0.0.1:8645 transfer \ "<founder mnemonic>" founder.tax.sov <some-test-account> 1
A node-accepted spend from founder.tax.sov is the end-to-end proof that the tax lands in an account you, and only you, control. Repeat for dev.tax.sov if you want both proven.
