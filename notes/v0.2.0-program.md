# v0.2.0 PROGRAM — PQ Shielded Pool v2, COMPLETE (no gaps, no after-the-fact)

**Mission**: v0.2.0 ships the ENTIRE post-quantum shielded pool: consensus-grade STARK
circuit → dormant consensus wiring → keys/addresses/wallet → RPC/CLI/KAT/SDK →
Station GUI pathways → activation machinery — every workstream enumerated HERE,
now. Anything not in this document that later turns out to be needed is a defect
of this document and gets a post-mortem, not a shrug.

**Owner directives (LAW for every slice):**
- SOLID AS GRANITE. NO MISTAKES, NO FAKE ANYTHING.
- No race conditions: concurrency only where a test proves ordering; shared state
  behind one lock discipline; no "probably fine" interleavings.
- No fuzzy logic: consensus paths use exact arithmetic, exact state machines,
  pinned KATs. No heuristics, no floating point, no "close enough".
- No made-up nothings: every claim is proven by a test/bench in the tree or it is
  not claimed. No mocked/simulated crypto anywhere in the trust path.
- Genesis `cb0272ff88e64c18cde0257f7fae1c8236b02651f10cc7a02456fd682ee2e72d` is
  FROZEN. Every slice proves byte-identity (genesis pin + KAT) before merge.
- Everything ships DORMANT behind a BIP-9 deployment. Activation is a separate,
  explicitly-approved event after external audit. The playbook is
  [The Upgrade Playbook](#w6--activation--upgrade-survivability) below.

**Foundation already on main (`f2cd519`)**: `chain/crates/shielded-pq` prototype
(real Winterfell STARK: commitment opening + depth-20 Merkle membership +
nullifier derivation + ownership binding in-circuit; ML-KEM-768 notes; ML-DSA-65
carrier auth; 21.2ms prove / 0.22ms verify / 35.5KB / 127-bit; 17/17 tests, KATs

## DECISIONS PINNED NOW (so nothing is relitigated mid-build)

| # | decision | value |
|---|----------|-------|
| D1 | Amount privacy | REQUIRED in-circuit for v0.2.0. Values are private in pool v2 at ship. No "public amounts for now". |
| D2 | Circuit arity | 4-in / 4-out per bundle (fixed shape, padded with dummy notes proven in-circuit to be zero-valued). Larger flows = multiple bundles. |
| D3 | Range checks | 64-bit in-circuit range check on every output value; sum checked in-circuit over a width that cannot wrap (values < 2^64, 4 inputs ⇒ sums < 2^66 held in field elements with explicit no-overflow argument). |
| D4 | Spend auth | ML-DSA-65 carrier signature over the canonical bundle digest binds {nullifiers, output commitments, anchors, value balance, fee, tx signer, nonce}. In-circuit sig verification is OUT of scope for 0.2.0 (documented trade-off: carrier auth is transparent-key-bound like today's pool; revisit in a future proof_version). |
| D5 | Anchors | Verifier accepts any of the last 128 v2 tree roots (anchor window), like Orchard's — prevents race between wallet witness and tip. |
| D6 | Proof system versioning | Every `ShieldedV2` bundle starts with a `proof_version: u8`. v0.2.0 = 1. Unknown version ⇒ clean consensus REJECT (never panic, never skip). Future circuits ship as proof_version N+1 behind a NEW deployment bit — no new Action variant needed. |
| D7 | Note detection | ML-KEM has no ECDH tag trick: wallets trial-decapsulate every v2 note. Each note ciphertext carries a 4-byte detection checksum = blake3(shared_secret ∥ "sov.v2.detect")[..4] so a failed decap is rejected in ~µs without AEAD work. Cost stated in runbook. |
| D8 | Address encoding | Unified address gains a v2 receiver under a NEW typecode in the existing UA container. UA parsing MUST ignore unknown typecodes (forward-compat law F3). Standalone v2 address = bech32m, HRP `xusq`. |
| D9 | HD derivation | v2 spend seed + ML-KEM keypair derive from the SAME 24-word phrase: domain-separated BLAKE3 subtree of the BIP-44 leaf (path documented + vector-tested in KAT). One phrase restores transparent + v1 + v2. |
| D10 | Tx size / weight | `ShieldedV2` carrier tx ≈ 40KB. Consensus tx-size cap and block data limits RAISED/AUDITED to admit ≥ 8 v2 bundles per block; mempool weight = bytes (proof included); fee floor scales by weight so v2 txs pay their space. Auction interplay: `Tipped{ inner: ShieldedV2 }` is LEGAL (tip rules unchanged); `ShieldedV2` may not wrap anything (it's a leaf action). |
| D11 | Drain limiter | Pool v2 gets its OWN de-shield drain limiter, same parameters as v1 (limit per 576-block window), own turnstile (v2 pool value can never go negative — consensus-enforced like v1). |
| D12 | Migration | v1→v2 via transparent hop only in 0.2.0 (unshield v1 → shield v2), chunked under both limiters, with amount-exposure disclosure in Station. A private bridge circuit is explicitly a FUTURE proof_version, not 0.2.0. |
| D13 | Activation | Deployment `shielded-v2`, **bit 2**, BIP-9 rules identical to bits 0/1 (period 288, threshold 9/10, LOT=false). Start/timeout heights chosen AT ARMING time (a later release bakes them, exactly like v0.1.99 did for bits 0/1). v0.2.0 ships with the deployment DEFINED in the registry but NOT armed (no baked start height). |
| D14 | SDK scope | TS SDK: full encode/decode + digest parity for `ShieldedV2` (KAT-pinned byte-for-byte) + detection/decap of incoming notes. Proving in TS is OUT (proofs are produced by Rust wallet/Station/CLI). |
| D15 | Winterfell hardening | `Proof::from_bytes` panic path is unacceptable in consensus: v0.2.0 wraps ALL deserialization in a catch-free pre-validator (length/domain checks before winterfell sees bytes) + fuzzing; if upstream still can panic on adversarial input, we vendor-patch or bound it. A panicking decode = consensus DoS = BLOCKER. |

## WORKSTREAMS (each = slices with acceptance proofs; nothing merges without them)

### W1 — Circuit completion (consensus-grade)
1. **S1a**: In-circuit value conservation for 4-in/4-out + 64-bit range checks +

   PRIVATE (public inputs shrink to: anchors, nullifiers, output commitments, net
   transparent value balance, fee).
2. **S1b**: Domain separation for every hash use (commitment/nullifier/owner/
   detection/digest each get distinct domains); KATs re-pinned; explicit
   no-collision-across-domains test.
3. **S1c**: Proof deserialization hardening per D15 + `cargo fuzz` targets for
   bundle decode and proof decode (corpus committed); zero panics under fuzz.
4. **S1d**: Parameter review to a written 128-bit target (FRI queries, blowup,

  clean; `-D warnings` clean.

### W6 — Activation + UPGRADE-SURVIVABILITY (the "runs in place forever" layer)
This is what makes v0.2.0 survive MULTIPLE future upgrades without retrofits:

**Forward-compatibility laws (enforced by tests that live in the tree forever):**
- **F1 — Deployment registry**: one table owns all signal bits (0 tx-domain,
  1 fee-auction, 2 shielded-v2, 3..28 reserved). Adding a deployment = one table
  row. Test: unknown/future bits set in a header are recorded and harmless.
- **F2 — proof_version ladder** (D6): new circuits ride the SAME Action with a
  version byte + new bit. Test: unknown proof_version ⇒ typed reject, no panic.
- **F3 — UA typecode tolerance**: parsers skip unknown receivers. Test: a UA
  containing a typecode from "the future" round-trips the known receivers.
- **F4 — schema_version ladder**: data-dir version checked at boot (exists since
  v0.1.9x); every state-shape change bumps it; downgrade = clean refusal, never
  corruption. Test: future-versioned data dir refuses cleanly.
- **F5 — RPC additive-only**: existing methods/fields never change meaning;
  capability discovery via presence. (The CLI already treats absent
  `sov_getShieldedInfo` as "older node" — that pattern is the law.)
- **F6 — Boot-order invariant** (v0.1.99's blocker, generalized): all consensus-
  affecting config installs before any replay. A single constructor owns chain
  construction; a test greps/asserts no bare construction bypasses it.
- **F7 — The Upgrade Playbook codified** in `chain/docs/upgrade-playbook.md`:
  additive+dormant → KAT/genesis byte-identity → Fable audit → release gate →
  CI-green tag → fleet deploy → signal → lock-in → activate behind grace.
  Every future fork follows this file or explains why in writing.

**Activation deliverable for 0.2.0 itself**: deployment row for bit 2 DEFINED but
unarmed (D13). Arming is a later 2-line release after the external audit, using
the identical baked-preset mechanism proven in v0.1.99.
