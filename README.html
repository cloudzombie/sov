The Mission

A sovereign-grade monetary system, built in the open

SOV is the Rust implementation of a reserve-asset Layer-1: auditable to the byte, with no hand-rolled cryptography and no fabricated data anywhere in the system. On a pure Nakamoto proof-of-work base it ships, as native protocol primitives, the pieces other ecosystems bolt on later — a Zcash-grade zk-SNARK shielded pool (Orchard/Halo2, no trusted setup), trustless cross-chain atomic swaps with Bitcoin and Zcash (HTLC, no custodian), post-quantum-native keys + transport (hybrid Ed25519+ML-DSA-65, ML-KEM-768), HD wallets (BIP-39/SLIP-10, Rust↔TS verified), native assets with issuer compliance, and a deterministic WebAssembly contract runtime.

No pre-mine

No premine, no investor allocation — every coin is minted by real proof-of-work under a fixed 21M cap. A perpetual 5%/2% founder/dev tax funds the project.

ASIC-resistant mining

RandomX (Monero's memory-hard PoW) lets commodity CPUs — Apple M-series included — mine fairly; the heaviest-work chain wins and finality is confirmation depth.

Provably valid

Exact-integer invariants (supply == genesis + mined), deterministic replay + cross-node agreement, heaviest-work fork-choice chaos/fault tests, KAT vectors, fuzzing, and reproducible builds.

Production Hardening

Privacy, trustless cross-chain, sound money — landed and proven

The most recent cycle drove SOV toward production as far as code allows. Every item below is implemented, tested with real cryptography, and green under the full gate (442 Rust + 118 SDK tests, fmt + clippy clean).

  • Zcash-grade shielded pool done — built on the audited Orchard/Halo2 stack with no trusted setup. Real zero-knowledge proofs for shielded transfers and for minting: the proof-of-work reward can be issued directly into the shielded pool (MineShielded), so coins are private from issuance. Note-commitment tree, nullifier set, and a canonical bundle codec all verify in-consensus.
  • Trustless cross-chain atomic swaps done — native HTLC primitives (HtlcLock/HtlcClaim/HtlcRefund) with SHA-256 hashlocks compatible with Bitcoin and Zcash OP_SHA256. The same preimage settles both sides, so swaps are atomic with no custodian, oracle, bridge, or liquidity pool.
  • HD wallets done — BIP-39 + SLIP-0010 (Ed25519) + BIP-44, with a cross-implementation known-answer test: the same mnemonic derives the same key in Rust and TypeScript, byte-for-byte.
  • Consensus timestamp hardening done — BIP-113 median-time-past plus a node-level future-time bound, closing timestamp-manipulation of difficulty.
  • Fixed-terminal-supply proof done — a conformance test proves emission terminates and that post-emission the total supply is fixed (no burn; fees are supply-neutral). Rationale in chain/docs/monetary-policy.md; the honest BIP mapping in chain/docs/bip-adoption.md.

Honest status. The above is the code-addressable path to production, and it is complete and green. Three things remain that no amount of code can close and that SOV will not claim without them: an external third-party security audit (especially of the new zk shielded-pool integration), a public testnet shakeout, and real Bitcoin/Zcash node operators running the bridge watchers. Until those land, SOV is a complete, auditable reference implementation — not a live mainnet.

Consensus

Pure Nakamoto proof-of-work. Only hashpower.

SOV's consensus is Bitcoin's, verbatim: blocks are sealed by proof of work, the heaviest-work chain wins, finality is confirmation depth, and the block coinbase is the only issuance path. There is no proof-of-stake, no validator committee, and no BFT — security is hashpower, and the mainnet seal is ASIC-resistant RandomX so commodity machines compete fairly.

The seal — ASIC-resistant RandomX

One algorithm secures the chain; whoever does the work produces the block.

  • RandomX (mainnet). Monero's memory-hard, CPU-optimized proof of work via the audited randomx-rs reference, keyed by the genesis hash — commodity machines (Apple M-series included) mine fairly and the chain resists ASIC capture. Sha256d is selectable for fast dev/test chains; the seal is a genesis-fixed PowAlgo meeting one 256-bit Target.
  • Committed difficulty (nBits). Each header carries its difficulty in Bitcoin's compact bits; a block cannot claim an easier target than the retarget rule requires.
  • Capped emission, no pre-mine. The coinbase pays 12.5 XUS halving every 840,000 blocks (~4 years at 2.5-minute blocks) under the 21M cap; genesis supply is exactly zero — every coin is mined.
  • Epoch retargeting. Difficulty retargets once per 2016-block epoch from the branch's actual-vs-expected timespan, Bitcoin-style (4×-clamped).

Fork choice & finality — Nakamoto

The most-work chain is the truth; settlement is depth, exactly like Bitcoin.

  • Heaviest-work fork choice. The active chain is the one with the most cumulative work (GetBlockProof in exact 256-bit math); a strictly heavier branch triggers a reorg.
  • Reorg correctness. A competing branch is replayed from the genesis ledger and adopted only if valid end-to-end, so an invalid heavier branch can never displace a valid head.
  • Confirmation-depth finality. A block is final once buried 6 deep (Bitcoin's convention) — probabilistic, and a pure function of chain state (survives restart). No votes to sync.
  • No validators. No stake, committee, proposer schedule, equivocation, or slashing exists — there is nothing to bribe or capture but hashpower.

How a transaction flows

1 · Submit

A wallet builds and signs a SignedTransaction. The mempool rejects it cheaply if the signature fails, the nonce is stale, or the pool is full — it does not check balances.

2 · Mine

Any node pulls a nonce-ordered batch, applies the coinbase + txs via sov_runtime, commits state, and grinds the header's nonce until its seal meets the difficulty. The header commits to the roots, the coinbase recipient, and the difficulty.

3 · Import

import_block runs the same validation on every block — peer or self-produced — checking the committed difficulty + proof of work, then re-executing against a clone of the ledger and refusing to commit unless the recomputed roots match.

4 · Confirm

Fork choice keeps the heaviest-work chain; the block accrues confirmations as the chain grows on top and is reported final at depth 6. No trusted path, no votes.

Execution enforces, in order: authentication (Ed25519 verifies against the tx pubkey) → authorization (that pubkey is the signer's registered key) → ordering (nonce equals the signer's current nonce) → execution. A nonce is consumed even when the inner action fails, so a rejected payment can never be replayed. Source: crates/runtime/src/execution.rs, crates/chain/src/blockchain.rs.

Height-keyed coinbase

MiningPolicy::reward_at(height, mined) pays 12.5 >> ((height − 1) / 840,000) SOV — Bitcoin's exact rule — clamped to the room left in the budget. The geometric series sums to 20,999,999.9076 XUS, strictly under the 21M cap; after the subsidy decays the chain runs on fees. Cadence: target_block_ms = 150,000 (2.5-minute blocks ⇒ ~4-year halvings, Zcash's cadence).

No pre-mine (founder/dev tax)

Because the budget equals the whole cap, a single grain of genesis balance or vesting makes genesis + budget > cap and is rejected — proven by mainnet_policy_forbids_any_premine. No premined treasury or investor allocation: every coin enters through proof of work. It is not a Bitcoin-style fair launch, though — a perpetual tax routes 5% to the founder and 2% to a dev fund out of every coinbase and every fee (the miner keeps 93%).

Emission is driven by one monotonic, state-root-committed counter — mined_emitted (crates/state/src/ledger.rs) — first-class consensus state because it cannot be recovered from balances once minted coins move. With checked Balance arithmetic throughout, total supply is supply == genesis + mined ≤ MAX_SUPPLY_GRAINS by construction — and on mainnet genesis == 0.

Governance

Hashpower decides upgrades — Bitcoin-style

SOV's on-chain governance (sov-governance) is the BIP-9 version-bits activation state machine plus BIP-8 mandatory activation. Miners signal readiness for a soft fork over signal windows; an upgrade locks in once a threshold of hashpower signals it. There is no stake-weighted or token-holder vote.

BIP-9 / BIP-8 threshold state machine

A deterministic state machine over miner-signal windows handles version-bits lock-in (BIP-9) with optional lock-in-on-timeout mandatory activation (BIP-8). Tested in isolation.

Why hashpower, not holders

Per design directive, governance is decided by the same actors who do the work of minting — never by stake or token-holder voting. On-chain autonomous "agents" were explicitly dropped as out of scope.

Integration boundary: wiring the signal bits into the live block header and the activation hooks is Phase 8 work. Source: crates/governance/src/lib.rs.

Verification & Security

A reserve asset cannot be "probably correct"

Every state transition, balance, and consensus decision must be provably valid, reproducible, and independently re-checkable. The sov-verify crate and the normative docs/state-transition.md make that real.

Exact-integer invariants

21M supply cap, value conservation (supply == genesis + mined), no unauthorized mint, and per-asset conservation — checked as exact u128 mathematics over real chain state.

Heaviest-work fork choice

Multi-replica chaos and fault-injection tests: nodes fed blocks in different orders converge on the most-work chain; an invalid heavier branch never displaces a valid head; confirmation depth only grows.

Deterministic replay

Independent nodes replay the same blocks to a byte-identical state_root at every height — ordered state, integer math, no wall-clock, no randomness.

Known-answer vectors

Committed KAT vectors (emitted by the real sov-katgen) re-derive Ed25519 keys/signatures, Blake3 ids, and Borsh bytes byte-for-byte — guarding against crypto/serialization drift.

Property tests & fuzzing

Proptest suites over random tx sequences ("decode arbitrary bytes never panics"), plus coverage-guided libFuzzer targets over the untrusted decode surface in chain/fuzz/.

Reproducible builds

Verified bit-for-bit: a clean script builds sov-node, sov-miner, and sov-katgen twice with identical SHA-256. Pinned Cargo.lock, single-unit codegen, fixed SOURCE_DATE_EPOCH, and a canonical container.

Node Operation

Build it, run it, mine it

Prerequisites: Rust stable ≥ 1.85 and git. The chain is pure Rust with std-only networking — no external system libraries required. The Cargo workspace lives under chain/.

Build & test

# clone, then enter the Rust workspace
git clone <repo> && cd sov/chain

# full workspace test + lint cycle
cargo test --workspace
cargo clippy --workspace --all-targets -- -D warnings
cargo fmt --all -- --check

Run a devnet & a real miner

# single-node transfer devnet — real chain, real signed txs, real hashes
cargo run -p sov-node

# real CPU SHA-256d miner — genuine PoW against the live difficulty,
# minting under the 21M cap. Release mode matters: mining is real work.
cargo run --release -p sov-node --bin sov-miner          # mine forever
cargo run --release -p sov-node --bin sov-miner 5        # mine 5 blocks

The node daemon & JSON-RPC

sov-rpc provides a long-running Daemon (the sov-rpcd binary) that boots from a JSON chain-spec, schedules block production, serves a std-only HTTP/1.1 + JSON-RPC 2.0 API, and persists to an append-only block log — resuming across restarts by deterministic replay through the validated import path (never trusting a snapshot).

RPC methods

health, chainId, getHeight, getSupply, getAccount, getBalance, getNonce, getBlock(ByHeight|ByHash), getBlockDigest, getHead, getStateRoot, getDifficulty, getMintReward, getMempoolSize, getConfirmations, isFinal, getMiners, and submitTransaction.

Binaries & clients

sov-rpcd (daemon) · sov-rpc-miner (external miner: reads the live head + difficulty, does real SHA-256d PoW against the per-miner challenge, submits the solution) · sov-wallet (balance + transfer) · sov-katgen (KAT-vector generator). P2P today does real TCP framing, transitive peer discovery, signed chain-bound handshakes, and catch-up sync; approval persistence/rebroadcast hardening is tracked in audit item R-02.

Reproducible builds

# build the release binaries twice; assert identical SHA-256
bash scripts/reproducible-build.sh

# or build inside the pinned container for machine-independent bytes
docker build -f chain/Dockerfile -t sov-build .   # FROM rust:1.93.1-slim
docker run --rm sov-build                          # prints SHA256SUMS (the attestation)

Continuous integration

CI runs on every push and pull request with six jobs: lint (fmt + clippy -D warnings), a build & test matrix across ubuntu-latest, macos-latest, and windows-latest (the clients are proven cross-platform on every commit), a named verify gate (cargo test -p sov-verify), a contracts job (wasm32 clippy + release build), a reproducible build job, and a nightly fuzz-targets build. Source: .github/workflows/ci.yml.

Roadmap

Built bottom-up, checked off as we go

The living build list lives in dashboard/phases.json; the Done lane only ever holds tasks verified against the actual repository. Phases 0–10 are complete — the only open item across the entire roadmap is Phase 7's external third-party audit + bug bounty, which requires a third party. Phase 21 — the Nakamoto migration — superseded several earlier phases The list below is kept for history. SOV is now pure proof-of-work: the Phase 1 Doomslug/BFT finality, Phase 2 staking, Phase 4 sharding, Phase 5 cross-chain pool / MPC, and Phase 6/19 compute marketplace were removed from the protocol (equihash was dropped for RandomX). Non-negotiable rules

  • No unsafe. #![forbid(unsafe_code)] on every one of the 28 crates, plus a per-crate [lints.rust] unsafe_code = "forbid".
  • No fabricated data, ever. Balances, miners, deposits, oracle prices, block counts, test counts — everything is derived from real state; empty or disconnected states are reported honestly.
  • No hand-rolled cryptography. Every primitive sits behind an audited crate.
  • Cross-platform clients. Node, miner, and wallet build and test on macOS, Windows, and Linux on every push.
  • Pure-Rust, std-only networking. Keeps the cross-platform promise cheap and the dependency surface auditable.
