★ Incremental shielded scanning

The old scan_pool replayed the entire chain from height 1 on every scan — O(chain) each time, unusable at real height. Replaced with NoteStore (chain/crates/shielded/src/store.rs): a persistent, encrypted note cache holding the scan height, wallet birthday, the note-commitment log, owned notes (raw Orchard parts + tree position + nullifier), and the spent-nullifier set. Each scan folds in only blocks scanned_height+1..=tip; the witness tree is rebuilt from the commitment log on load (bridgetree has no serde), so witnesses match consensus exactly. Cache is keyed by the wallet's implicit id and encrypted at rest (ChaCha20-Poly1305, device key) at ~/.sov-station/notes/<id>.store. A tip below the cached height (reset/deep reorg) rebuilds from genesis.

live, against the running node @ height 1616:
COLD full scan  : 1616 blocks → 2.28 s
WARM re-scan   : 1 new block → 2.4 ms  (~950× faster)
identical balance (1,677,540,847,118 grains · 4 notes) · cache 1827 bytes
crate tests: incremental==full + survives serialize + spend-after-reload ✓

2

Reorg-aware note rollback

A scan assumed a linear chain; a reorg (or a chain reset) could leave the cache extending an orphaned branch — stale notes, wrong positions, bad witnesses. Now NoteStore records a per-block checkpoint (canonical header hash + the commitment/owned/spent log lengths at that height), a rolling window of the last REORG_HORIZON = 100 blocks (well beyond the 6-conf finality depth; matches Zcash's reorg limit). Before each scan, scan_store checks the node's hash at the cached tip; on a mismatch it walks the checkpoints newest→oldest to find the deepest height that still agrees (the fork point) and rollback_to(fork) — an O(1) truncation of every log to the recorded lengths, then the witness tree is rebuilt. A reorg deeper than the window rebuilds from the wallet birthday. The next scan resumes at fork+1, cleanly undoing the orphaned branch's notes and spends.

crate tests (real Halo2):
• reorg rolls back to fork, rescans new branch, == a fresh full scan ✓
• witnessing works after rollback+rescan (tree rebuilt twice) ✓
• survives serialize round-trip with checkpoints ✓
• checkpoint window bounded to 100; deep reorg ⇒ rebuild ✓
also fixed: a chain reset (tip drops below cache) now reconciles instead of silently corrupting.

★ Multisig / threshold signatures

Opt-in M-of-N control for any account (consensus, done + tested). SetMultisig{signers,threshold} converts an account to M-of-N — once set, single-key spends are disabled and every action must go through MultisigExec{action, approvals}, which requires threshold distinct signer approvals over a canonical (account, nonce, action) message (so approvals are bound to the exact op — no replay, no cross-action reuse). The inner action runs through the same validated dispatch as any tx. Built additively (policy in an absent-when-empty map, actions appended) so the genesis hash is byte-identical — no chain reset needed, your testnet keeps running.

tests: 2-of-3 enforces threshold & disables single-key · duplicate/misbound approvals rejected · policy commits to root + survives save/load ✓
genesis KAT byte-identical · mempool admission gate mirrors the auth · N≤32 signers.

Watch-only + offline (air-gapped) signing

Air-gapped signing (the export→sign→broadcast split): a "🔌 Offline / air-gapped signing" panel — build unsigned here (works from watch-only; pulls the on-chain nonce), sign on the machine that holds the seed (no network; the key must match the tx, enforced by SignedTransaction::sign), broadcast from an online node. All signing paths are gated so a watch-only wallet can't attempt to sign. Wallet-side only — no chain change.

★ Node-data stability across upgrades

The node resumes by replaying its block log (state is re-derived, never trusted from a snapshot) — so additive changes (new actions, new state, RPC) resume fine. Hardened: a schema_version file gates the data dir (an incompatible binary errors clearly instead of mis-replaying), and a checksum-valid-but-undecodable record now fails loudly instead of silently truncating a persisted chain (a torn crash-tail is still safely dropped).

8

Mempool fee economics

Audit: SOV uses a fixed gas price (no fee auction), so there is no per-tx bid to prioritize by — the mempool's job under pressure is fairness + anti-DoS, not fee-bidding. Defined & built: a per-sender cap (one account may hold ≤ 1/64 of the pool, floored at 16) so no account can crowd others out; and eviction under pressure — a full pool evicts the most over-represented sender's highest-nonce (least-executable) tx to admit a newcomer, while a fairly-shared full pool rejects rather than thrash. (A true fee market would need a per-tx tip field — a consensus/encoding change, deferred so the persistent testnet isn't disrupted.)

tests: per-sender cap bounds one account · full pool evicts heaviest sender's highest nonce · fair full pool rejects (no thrash) ✓ · in-memory only (chain-safe)
